# Rocq Formal Verification

This directory contains Rocq (formerly Coq) translations of the `encore_vm`
Rust crate, generated by [rocq-of-rust](https://github.com/formal-land/rocq-of-rust),
along with the build infrastructure to compile them.

## Prerequisites

- **Rocq 9.1** (installed via opam, e.g. in a dedicated switch)
- **dune >= 3.21** (comes with the opam switch)
- The `rocq-stdlib` opam package (provides `Stdlib` and `Ltac2` theories)

No other Rocq libraries are required — the necessary parts of the
`RocqOfRust` support library are vendored locally.

## Directory Layout

```
rocq/
├── dune-project                 # Dune project root (rocq lang 0.11)
├── RocqOfRust/                  # Vendored RocqOfRust support library
│   ├── dune                     # Theory config: depends on Stdlib, Ltac2
│   ├── M.v                      # Rust monad definition (types, monadic ops)
│   ├── RecordUpdate.v           # Record update tactics (Ltac2-based)
│   ├── RocqOfRust.v             # Top-level re-export module
│   └── lib/
│       └── lib.v                # Auxiliary definitions and notations
└── encore_vm/                   # Generated Rocq translation of the VM
    ├── dune                     # Theory config: depends on RocqOfRust
    ├── arena.v                  # Bump allocator (Arena struct + alloc/read/write)
    ├── code.v                   # Bytecode instruction container
    ├── error.v                  # VM error types (HeapOverflow, etc.)
    ├── gc.v                     # Mark-compact garbage collector (Lisp 2)
    ├── opcode.v                 # Opcode enum definitions
    ├── program.v                # Program representation (code + globals)
    ├── value.v                  # Packed 32-bit Value type and accessors
    └── vm.v                     # Main VM execution loop
```

## Building

```bash
cd rocq/
eval $(opam env)      # ensure dune/rocq are on PATH
dune build
```

This compiles all `.v` files into `.vo` objects under `_build/`.

## Re-generating the `.v` files

If the Rust source in `crates/encore_vm/` changes, regenerate the
Rocq translations:

```bash
cd crates/encore_vm/
rustup run nightly-2024-12-07 cargo rocq-of-rust
mv src/*.v ../../rocq/encore_vm/
```

Then rebuild with `dune build` from the `rocq/` directory.

## Vendored RocqOfRust Library

The `RocqOfRust/` directory contains a minimal subset of the
[RocqOfRust](https://github.com/formal-land/rocq-of-rust/tree/main/RocqOfRust)
support library — only the four files needed to compile the generated code:

- `M.v` — The Rust monad (`M`), value/type representations, and monadic
  operations (`M.alloc`, `M.read`, `M.write`, `M.call_closure`, etc.)
- `RecordUpdate.v` — Generic record update tactic infrastructure
- `lib/lib.v` — Auxiliary Rocq definitions and notations
- `RocqOfRust.v` — Facade module that re-exports everything above

### Patches applied for Rocq 9.1 compatibility

The upstream RocqOfRust targets Rocq 9.0. Two changes were made to
compile under Rocq 9.1:

1. **Removed external dependencies** from `M.v`: the `Require Export` lines
   for `Hammer.Tactics`, `smpl.Smpl`, and `coqutil.Datatypes.List` were
   stripped. These are proof-automation libraries not needed for compilation
   of the generated code.

2. **Added `-type-in-type`** to the RocqOfRust theory flags. Rocq 9.1
   enforces stricter universe constraints than 9.0, and the `sigS` and
   `Pointer.Core.t` inductives quantify over `Type`/`Set` in ways that
   require this flag. This is safe for our use case since universe
   consistency is not relied upon in the generated translations.

The `encore_vm` theory itself does **not** use `-type-in-type` — only the
vendored support library needs it.
